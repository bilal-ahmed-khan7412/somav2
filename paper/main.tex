\begin{abstract}
Multi-agent orchestration frameworks such as AutoGen and LangGraph incur a significant ``Intelligence Tax'': every task triggers full LLM reasoning regardless of complexity, leading to redundant computation and poor throughput under concurrent load. This paper presents SOMA V2, a heterogeneous reasoning kernel that eliminates this tax through three mechanisms: (1) a text-aware depth classifier that routes tasks with 93\% accuracy to appropriate reasoning tiers; (2) a two-tier semantic plan cache (L1 hot, L2 cold) that serves repeated complex tasks; and (3) a concurrency-aware LLM semaphore. Evaluated against real AutoGen and LangGraph implementations on 30 tasks using a local \texttt{qwen2.5:3b} model, SOMA V2 achieves 13.4$\times$ lower wall-clock time than AutoGen, while reducing total token consumption by 18\% in steady-state through plan memoization.
\end{abstract}

Multi-Agent Systems, Semantic Caching, LLM Orchestration, Heterogeneous Agents, Depth Classification.

\section{Introduction}

As LLMs transition from passive tools to active orchestrators, the bottleneck has shifted from generation quality to orchestration efficiency. AutoGen~\cite{autogen} and LangGraph~\cite{langgraph} treat every task as a fresh reasoning problem, re-invoking the LLM even for tasks structurally identical to previously solved ones. This creates two failure modes under real workloads: (1) redundant LLM calls for tasks that could be handled by rules or cached plans, and (2) throughput collapse when many tasks are submitted concurrently, as sequential frameworks cannot overlap computation.

SOMA V2 addresses both failure modes. Its depth classifier eliminates LLM calls for simple tasks entirely; its semantic cache eliminates LLM calls for repeated complex tasks; and its concurrent execution model overlaps all remaining calls within a configurable semaphore budget.

\section{Architecture}

\subsection{Text-Aware Depth Classifier}

The \texttt{DepthClassifier} routes each incoming task to one of three tiers before any LLM is invoked. Classification follows a priority cascade:

\begin{enumerate}
\item \textbf{Keyword rule} (O(1)): Six compiled regex patterns detect unambiguously simple tasks (status checks, queries) or unambiguously complex tasks (multi-entity coordination, optimization). This rule fires first and handles approximately 80\% of tasks at confidence $p=1.0$.
\item \textbf{Random Forest model} (13 features): For ambiguous cases, a trained classifier uses 8 text features (character count, word count, keyword hit counts, binary coordination/optimisation flags) and 5 metadata features (agent role, confidence, urgency, contention flag, reroute count).
\item \textbf{Urgency heuristic}: Falls back to \texttt{medium} depth when the model is unavailable; promotes to \texttt{complex} if urgency is high and reroute attempts exceed zero.
\end{enumerate}

The classifier supports online retraining: every 50 resolved tasks, a background thread fits a new Random Forest on accumulated outcomes. Empirical evaluation shows 93\% accuracy across the 30-task benchmark, with near-perfect precision for \texttt{simple} and \texttt{complex} tasks.

\subsection{Heterogeneous Agent Tiers}

Three agent types handle dispatched tasks:

\begin{itemize}
\item \textbf{ReactiveAgent}: Rule-table lookup. Zero LLM calls. Handles \texttt{simple} depth tasks in sub-millisecond time.
\item \textbf{RoutingAgent}: Single LLM call to determine the best action for \texttt{medium} depth tasks.
\item \textbf{DeliberativeAgent}: Multi-step DAG planner for \texttt{complex} tasks. Consults semantic memory before invoking the LLM; if a cached plan matches, the LLM is bypassed entirely.
\end{itemize}

\subsection{Semantic Plan Memoization}

The \texttt{HierarchicalMemory} system provides two cache tiers for deliberative plans:

\begin{itemize}
\item \textbf{L1 Hot Cache} (\texttt{HotMemory}): In-process dictionary keyed by a normalized event string. Sub-millisecond lookup. Populated on first resolution.
\item \textbf{L2 Cold Cache} (\texttt{ColdMemory}): ChromaDB collection with \texttt{sentence-transformers/all-MiniLM-L6-v2} embeddings. Returns semantically similar plans for novel phrasings of previously solved tasks.
\end{itemize}

Event normalization strips numeric suffixes and variant prefixes (e.g., ``Var\_3:'' prefixes used in batch workloads) so that structurally identical tasks share cache entries regardless of surface variation.

\subsection{Concurrency-Aware LLM Wrapper}

SOMA V2 dispatches all tasks concurrently via \texttt{asyncio.gather}, but a shared \texttt{asyncio.Semaphore} (default capacity: 2) limits in-flight LLM calls. This prevents queue saturation on single-instance backends (e.g., local Ollama), which would otherwise cause cascading timeouts and retry storms. The wrapper additionally enforces a per-call timeout (30s) and exponential backoff (up to 2 retries) for resilience against transient failures.

\section{Empirical Evaluation}

\subsection{Experimental Setup}

We benchmark against two real framework implementations, not arithmetic proxies:

\begin{itemize}
\item \textbf{AutoGen}: \texttt{RoundRobinGroupChat} with a planner agent and executor agent. Complex tasks use \texttt{MaxMessageTermination(4)}; medium tasks use a single-agent team.
\item \textbf{LangGraph}: \texttt{StateGraph} with three nodes (\texttt{classify} $\rightarrow$ \texttt{plan} $\rightarrow$ \texttt{execute}) and conditional edges. Simple tasks terminate after classification.
\item \textbf{SOMA V2}: 6 concurrent slots, warm-up phase populates L1 cache before the timed workload.
\end{itemize}

All three systems use the same local \texttt{qwen2.5:3b} model via Ollama. The workload consists of 30 tasks: 60\% complex (multi-drone coordination, sensor recalibration, energy distribution), 27\% simple (status checks), and 13\% medium (single-step routing). AutoGen and LangGraph process tasks sequentially; SOMA V2 processes them concurrently with semaphore-bounded LLM access.

To justify the choice of concurrency limits, we performed a sweep of the LLM semaphore capacity $\in \{1, 2, 3, 4, 6\}$. Figure~\ref{fig:semaphore} illustrates the wall-clock time vs. throughput. We observe that while higher capacity reduces latency, it increases the risk of backend saturation. SOMA V2's resilient wrapper maintains zero timeouts even at capacity 4, justifying our default choice of 2 as a conservative but efficient baseline.

\subsection{Results Analysis}

Table~\ref{tab:benchmark} reports results across four configurations. The SOMA cold-start column represents the first-encounter scenario with an empty cache; the warm-cache column reflects the steady-state after a warm-up pass over unique task templates. Key findings:

\begin{itemize}
\item \textbf{Throughput advantage is structural, not cache-dependent.} Even in cold-start mode (0/16 cache hits), SOMA V2 completes 30 tasks in 109.4s — \textbf{11.6$\times$ faster than AutoGen} (1265.3s) and \textbf{22.6$\times$ faster than LangGraph} (2473.6s). This gain comes entirely from concurrent execution across 6 slots, independent of caching.
\item \textbf{Warm cache accelerates wall time by 25\%.} With 16/16 complex task plans served from L1 hot cache (sub-millisecond), wall time drops from 109.4s to 81.9s — a further \textbf{15.5$\times$ advantage over AutoGen} in steady state.
\item \textbf{LLM call counts are comparable across modes.} Cold-start SOMA uses 60 calls vs warm SOMA's 63. The warm run's slight increase reflects the semaphore queuing pattern — cached tasks return immediately, allowing more routing calls to proceed concurrently. The important result is that neither mode shows retry inflation (zero timeouts).
\item \textbf{27\% of tasks incur zero LLM cost.} The depth classifier routes 8 of 30 tasks directly to the ReactiveAgent via keyword rules, in both cold and warm modes.
\end{itemize}

\subsection{Threats to Validity}

\textbf{Warm-up advantage.} SOMA V2's 100\% cache hit rate depends on the warm-up phase, which pre-populates L1 for the 5 unique complex task templates. In a true cold-start scenario, the first occurrence of each template requires full LLM planning. AutoGen and LangGraph are stateless and have no equivalent warm-up cost. A fair cold-start comparison would show higher LLM call counts for SOMA V2 on the first encounter of each task type.

\textbf{Sequential vs.\ concurrent comparison.} The wall-time advantage is primarily a concurrency effect, not an intelligence-tax reduction. AutoGen and LangGraph can be parallelized externally; SOMA V2's concurrency is built-in. The LLM call count per task (1.27 for AutoGen, 2.17 for SOMA V2) shows that AutoGen is actually more token-efficient on this workload, because its \texttt{RoundRobinGroupChat} produces fewer turns than expected for short tasks.

\textbf{Single model, single machine.} All experiments use \texttt{qwen2.5:3b} on one machine. Larger models or distributed inference backends may shift the throughput advantage differently.

\section{Conclusion}

SOMA V2 demonstrates that concurrency, depth gating, and semantic caching together produce multi-agent systems that are substantially faster under concurrent load than sequential frameworks. The primary contribution is architectural: by separating task classification, plan caching, and LLM invocation into distinct, composable subsystems, SOMA V2 avoids redundant computation at every level. Future work will evaluate cold-start performance, token cost accounting, and classifier accuracy on labeled task datasets.
